# Chapter 6 - 13: Engine Guide

## Purpose
This document provides a deep dive into every non-RAG engine within the Mobiloitte AI Knowledge Assistant. While the RAG engine handles document search, these smaller, specialized engines form the rigid outer perimeter of the system. They intercept malicious queries, normalize text, resolve acronyms, make routing decisions, and serve instant deterministic answers for static intents like greetings or company policies.

---

## 1. Validation & Pre-Processing Engines

### 1.1 Validation Engine (`app/engines/validation.py`)
- **Purpose:** The first line of defense. It prevents denial-of-service (DoS) via massive strings and blocks basic injection attacks.
- **How it works:** It checks the raw query length. If `len(query) > 1000`, it immediately sets `handled=True` and returns a rejection message.
- **Why this design was chosen:** Catching garbage input *before* the Decision Engine or the Embedding Model saves CPU cycles and protects the database.

### 1.2 Normalization Engine (`app/engines/normalization.py`)
- **Purpose:** Standardizes input so downstream regex and keyword matchers don't fail due to casing or punctuation.
- **How it works:** It strips leading/trailing whitespace, forces lowercase, and removes excess special characters. 
- **Input:** `"  WhAt is the LEAVE POLICY!?!  "`
- **Output:** `"what is the leave policy"`
- **Note:** It does *not* set `handled=True`. It modifies `context.metadata["resolved_query"]` and passes the context to the next engine.

### 1.3 Alias Engine (`app/engines/alias.py`)
- **Purpose:** Resolves corporate acronyms and synonyms into their full canonical forms to dramatically improve vector search accuracy.
- **How it works:** Uses `configs.alias.ALIAS_MAP`. If the query contains "WFH", it rewrites it to "Work From Home". 
- **Why it exists:** An embedding for "WFH" and "Work From Home" might end up in different vector spaces. Normalizing them ensures high recall during vector search.

## 2. The Decision Engine (`app/engines/decision.py`)

- **Purpose:** To classify the query's intent and make a routing decision. It acts as a traffic cop.
- **How it works:** It scans the normalized query against a prioritized list of regular expressions and keyword clusters defined in `app/configs/`. 
- **Intents Detected:**
  - `GREETING` (e.g., "hello", "hi")
  - `GOODBYE` (e.g., "bye", "see you")
  - `THANKS` (e.g., "thanks", "thank you")
  - `SMALL_TALK` (e.g., "how are you", "what is your name")
  - `FAST_PATH` (e.g., "who is the ceo", "where is the office")
- **Fallback Rule:** If none of the static intents match, it defaults to `RAG` (Retrieval-Augmented Generation), assuming the user is asking a business question that requires document search.
- **Output:** Sets `context.intent` and stores the routing decision in `context.metadata`.

---

## 3. Static Response Engines

These engines listen for specific intents. If their target intent is detected, they execute, serve a predefined answer, and abort the rest of the pipeline (`handled=True`).

### 3.1 Greeting Engine
- **Purpose:** To handle pleasantries without querying the database or LLM.
- **Flow:** If `context.intent == "GREETING"`, it returns a random greeting from a list (e.g., "Hello! I am the Mobiloitte Enterprise Assistant. How can I help you today?").
- **Edge Cases:** What if the user says "Hello, what is the leave policy?". The Decision Engine is smart enough to see "leave policy" and assign the `RAG` intent instead of `GREETING`, bypassing the Greeting Engine completely.

### 3.2 Goodbye Engine & Thanks Engine
- **Purpose:** To close out conversations gracefully.
- **Flow:** Returns strings like "You're welcome!" or "Goodbye!".

### 3.3 Small Talk Engine
- **Purpose:** To handle identity questions.
- **Flow:** If the user asks "Are you AI?" or "Who made you?", it returns a strict, non-hallucinated definition: "I am the Mobiloitte Enterprise AI Assistant..."

## 4. The FastPath Engine (`app/engines/fast_path.py`)

- **Purpose:** To provide instant, deterministic, hardcoded answers for critical company facts that should *never* be generated by an LLM.
- **How it works:** It reads `app.configs.validation.FAST_PATH_RULES`. If a query perfectly matches a fast-path trigger (e.g., "what is the company name"), it returns the hardcoded answer.
- **Why FastPath exists:** If a CEO asks the bot "Who is the CEO?", and the LLM hallucinates the wrong name because it misread an old document, it's a catastrophic failure. FastPath guarantees 100% accuracy for critical facts.
- **Document-Aware Delegation:** A critical requirement implemented during the hardening phase. FastPath used to steal queries like "services" or "policies". Now, FastPath is *document-aware*. Before it answers, it checks if the query heavily overlaps with uploaded document knowledge. If the user asks about "services", and a document named "Services_Guide.pdf" is uploaded, FastPath skips and delegates to RAG.

---

## 5. The Knowledge Engine (`app/engines/knowledge.py`)

- **Purpose:** To bridge the gap between deterministic routing and dynamic RAG.
- **How it works:** It verifies if the system is currently connected to the vector database and checks if any documents are actually indexed.
- **Flow:** If the Decision Engine says the query needs RAG, but the Knowledge Engine detects that the database is empty (no documents uploaded), it short-circuits the pipeline and returns: "I cannot answer that because no documents have been uploaded to my knowledge base."
- **Why this design was chosen:** It prevents the RAG engine from executing expensive, doomed vector searches when the database is empty.

---

## 6. The Fallback Engine (`app/engines/fallback.py`)

- **Purpose:** The safety net. It runs *last* in the pipeline. If a query reaches this engine, it means every other engine (including RAG) either failed or refused to handle the query.
- **How it works:** It simply returns a polite "I'm sorry, I don't understand that query or I couldn't find enough information in the documents."
- **Common Scenarios:** 
  1. The user types gibberish ("asdfasdf").
  2. The LLM API goes down and the RAG engine fails completely.
  3. The user asks an out-of-domain question ("How do I bake a cake?") and the Evidence Gate inside the RAG engine blocks it. The request falls through to the Fallback Engine.

## 7. Manager Interview Questions

**Q: Why don't we use Gemini to answer "Hello" or "How are you"?**
*Answer:* Two reasons: Cost and Latency. A FastPath/Greeting engine evaluates in 0.05 milliseconds and costs $0.00. Calling Gemini to respond to "Hello" takes 800 milliseconds and consumes tokens. Over a million queries, serving static intents locally saves massive infrastructure costs and provides a much snappier user experience.

**Q: How do we prevent the AI from saying the wrong CEO name?**
*Answer:* Through the FastPath Engine. We intercept the query using regex *before* it reaches the database or the LLM, and we return a hardcoded, deterministic string. The LLM is physically incapable of hallucinating the CEO's name because the request never reaches it.
