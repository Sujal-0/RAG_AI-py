# Chapter 18: Database Design

## Purpose
The database is the foundation of the RAG system. It must store relational metadata (documents, chat history, user sessions) alongside high-dimensional mathematical arrays (vectors) and query them simultaneously.

This chapter details the PostgreSQL schema, the `pgvector` extension, and the mechanics of Hybrid Search.

---

## 1. Why PostgreSQL and `pgvector`?

Historically, AI applications required two databases:
1. A Relational DB (like MySQL or Postgres) to store user accounts, chat logs, and document titles.
2. A Vector DB (like Pinecone, Milvus, or Qdrant) to store the vector embeddings for semantic search.

This "split-brain" architecture is notoriously difficult to maintain. When a document is deleted, you must remember to delete it from both databases. If one database goes down, the system state becomes corrupted.

**`pgvector` solves this.** It is an open-source extension for PostgreSQL that allows Postgres to natively store and query vector types alongside standard relational data.
- **Single Source of Truth:** Documents, chunks, and vectors live in the same transaction space.
- **ACID Compliance:** If a document upload fails halfway, the entire transaction rolls back cleanly.
- **Hybrid Search:** We can perform a vector similarity search *and* filter by a relational column (like `document_id`) in a single SQL query. This is impossible in a split-brain architecture without massive network latency.

## 2. Table Schemas

The database consists of three primary tables defined in `app/models/` via SQLAlchemy.

### 2.1 `documents` Table
Stores the high-level metadata for an uploaded file.
- `id` (UUID): Primary Key.
- `filename` (String): The name of the uploaded file (e.g., `HR_Policy.pdf`).
- `content_type` (String): e.g., `application/pdf`.
- `status` (String): e.g., `PROCESSING`, `READY_FOR_SEARCH`, `FAILED`.
- `created_at` (DateTime): Upload timestamp.

### 2.2 `chunks` Table (The Core)
Stores the individual paragraphs of text and their corresponding vector embeddings.
- `id` (UUID): Primary Key.
- `document_id` (UUID): Foreign Key linking back to `documents`.
- `text` (Text): The raw text of the paragraph.
- `page_number` (Integer): Which page the chunk came from (for citations).
- `heading` (String): The detected heading under which this chunk falls.
- **`embedding` (Vector(384)):** The 384-dimensional floating-point array generated by MiniLM.

### 2.3 `chat_history` Table
Stores conversation logs for memory context.
- `id` (UUID): Primary Key.
- `session_id` (String): Grouping identifier for a conversation.
- `role` (String): `user` or `assistant`.
- `content` (Text): The actual message text.
- `created_at` (DateTime): Timestamp for ordering.

## 3. Database Indexes

Vectors are just long lists of numbers. A naive vector search compares the query vector to *every single row* in the database to find the closest match. This is called Exact Nearest Neighbor (KNN). It is 100% accurate but becomes painfully slow after a million rows.

To solve this, we use Approximate Nearest Neighbor (ANN) indexes.

### HNSW (Hierarchical Navigable Small World)
We create an index on the `embedding` column using HNSW and the `vector_cosine_ops` operator class.
- **How it works:** It builds a multi-layered graph of vectors, linking closest neighbors. During a search, it navigates the top layers to quickly zoom in on the neighborhood of the query vector, then drops down to the detailed layer.
- **Why this design was chosen:** HNSW is slightly slower to build during ingestion than IVFFlat, but it provides significantly faster query times and much higher recall (accuracy) without needing to be retrained as data grows.

## 4. Hybrid Search Mechanics (Under the Hood)

When a user asks a question, the `KnowledgeEngine` executes a complex SQL query. It isn't just one search; it's a CTE (Common Table Expression) combining three distinct algorithms.

1. **Vector Search (Cosine Distance):**
   ```sql
   SELECT id, 1 - (embedding <=> :query_vector) AS vector_score 
   FROM chunks
   ORDER BY embedding <=> :query_vector LIMIT 10
   ```
   *Note: `pgvector` uses `<=>` to represent cosine distance. `1 - distance` gives us the cosine similarity score (0.0 to 1.0).*

2. **Keyword Search (BM25 or ILIKE):**
   ```sql
   SELECT id, 1.0 AS keyword_score 
   FROM chunks 
   WHERE text ILIKE :keyword
   ```

3. **Metadata / Heading Search:**
   ```sql
   SELECT id, 1.0 AS metadata_score 
   FROM chunks 
   WHERE heading ILIKE :keyword
   ```

**Reciprocal Rank Fusion (RRF):**
The results are joined. RRF assigns a score based on the rank of the item in each individual search result. A chunk that is Rank #1 in Vector and Rank #2 in Keyword gets a massively boosted RRF score, ensuring the absolute most relevant chunks rise to the top.

## 5. Manager Interview Questions

**Q: If we upload 100,000 documents, will the database crash during a search?**
*Answer:* No. Because we use PostgreSQL with HNSW indexing on the `pgvector` column, query latency remains in the single-digit milliseconds even with millions of rows. The HNSW graph allows the database to skip 99% of the rows and only calculate distances for vectors in the immediate "neighborhood" of the query.

**Q: Can a user search for information across all documents, or only specific ones?**
*Answer:* Both. Because we use a unified architecture (relational + vector in one database), we can easily append a `WHERE document_id = X` clause to our vector search. If the user selects a specific folder in the UI, the database effortlessly filters the search space before executing the semantic search, a massive advantage over standalone vector databases.
